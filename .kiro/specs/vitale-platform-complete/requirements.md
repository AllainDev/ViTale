# Requirements Document

## Introduction

ViTale is an AI-powered travel platform that combines physical handcrafted wool dolls with a digital "AI Travel Passport" to create an immersive cultural storytelling experience. The platform uses QR code activation to connect physical products with a 3D digital assistant that delivers location-based cultural narratives through a gamified checkpoint system. The solution includes B2B integrations with restaurants, hotels, and tour operators to create a complete travel ecosystem.

The platform operates on minimal infrastructure (single VPS <$10/month) using Clean Architecture principles, with a static frontend deployed to CDN for optimal performance and offline-first capabilities for uninterrupted user experience.

## Glossary

- **ViTale_Platform**: The complete system including frontend PWA, backend API, database, and 3D rendering engine
- **Traveler**: A user of the platform who may be anonymous or registered
- **AI_Travel_Passport**: Digital passport showing stamps, badges, unlocked stories, and travel progress
- **QR_Product**: Physical wool doll or passport cover with unique QR code for activation
- **Character**: 3D digital representation of regional cultural figure tied to a QR_Product
- **Story_Chapter**: Segment of cultural narrative unlocked by visiting checkpoints
- **Checkpoint**: Geographic location (GPS coordinates) that unlocks story content when visited
- **Anonymous_ID**: Temporary identifier for users before account registration (lazy registration pattern)
- **Checkin_Record**: Record of Traveler visiting a Checkpoint, supporting offline creation
- **Stamp**: Visual achievement marker earned by visiting locations
- **Badge**: Achievement award for completing specific activity patterns
- **Voucher**: Discount or offer from Partner businesses claimable by Travelers
- **Partner**: B2B entity (restaurant, hotel, tour operator) participating in the ecosystem
- **Action_Tag**: Instruction embedded in AI response to trigger 3D character animations (e.g., [WAVE], [SMILE])
- **Chat_Session**: Conversation thread between Traveler and AI assistant
- **Condensed_Context**: Compressed conversation history generated by Groq for token efficiency
- **Sync_Worker**: Background process that uploads offline checkins when network available
- **IndexedDB**: Browser storage for offline-first data persistence

- **GLB_File**: 3D model file format for characters, compressed with draco/meshopt
- **Service_Worker**: Browser API enabling offline-first functionality through caching
- **Docker_Compose**: Container orchestration tool for deploying all platform services
- **HTTP_Only_Cookie**: Secure cookie that prevents JavaScript access, used for JWT storage
- **SSG**: Static Site Generation, Next.js mode for pre-rendering pages
- **PWA**: Progressive Web App supporting installation and offline functionality
- **Background_Sync**: Browser API for deferred network requests when connectivity restored
- **Clean_Architecture**: Design pattern separating Domain/Application/Infrastructure/Presentation layers
- **JSONB**: PostgreSQL data type for storing flexible JSON with indexing capability
- **Rate_Limiter**: Middleware preventing abuse by limiting requests per time window
- **Idempotent_Operation**: API operation that produces same result when called multiple times
- **CDN**: Content Delivery Network (Cloudflare) for serving static assets globally
- **VPS**: Virtual Private Server hosting the containerized backend
- **JWT**: JSON Web Token for stateless authentication
- **OAuth_Provider**: External authentication service (Apple, Google)

## Requirements

### Requirement 1: Infrastructure Foundation

**User Story:** As a DevOps engineer, I want a containerized deployment environment with automated CI/CD, so that the platform can run reliably on minimal infrastructure with predictable costs.

#### Acceptance Criteria

1. THE ViTale_Platform SHALL run on a single VPS with 2 vCPU and 2-4GB RAM
2. THE Docker_Compose configuration SHALL define three services: api, db, and web
3. THE db service SHALL use postgres:16-alpine image with persistent volume storage
4. THE api service SHALL expose health check endpoint at /health returning HTTP 200
5. THE web service SHALL expose health check endpoint at / returning HTTP 200
6. WHEN docker compose up executes, THEN THE ViTale_Platform SHALL start all services within 60 seconds
7. THE private network vitale_net SHALL isolate service communication from external access
8. THE .env.example file SHALL document all required environment variables: DB_CONNECTION_STRING, JWT_SECRET, GROQ_API_KEY, COOKIE_DOMAIN
9. WHERE a GitHub Actions workflow is configured, THE CI pipeline SHALL build and test all services on pull request
10. WHERE a GitHub Actions workflow is configured, THE deployment pipeline SHALL deploy to VPS via SSH on merge to main branch

### Requirement 2: Domain Configuration

**User Story:** As a developer, I want DNS and CDN properly configured with SSL certificates, so that users can access the platform securely through branded subdomains.

#### Acceptance Criteria

1. THE domain vitale.vn SHALL resolve to the VPS IP address
2. THE subdomain app.vitale.vn SHALL serve the frontend application
3. THE subdomain api.vitale.vn SHALL serve the backend API
4. WHEN a client requests https://app.vitale.vn, THEN THE CDN SHALL serve the request with valid SSL certificate
5. WHEN a client requests https://api.vitale.vn, THEN THE CDN SHALL serve the request with valid SSL certificate
6. THE Cloudflare proxy SHALL cache static assets matching patterns: /assets/*.glb, *.png, *.mp3, *.jpg, *.woff2
7. THE cached static assets SHALL have a TTL of at least 86400 seconds (24 hours)
8. THE Cloudflare compression SHALL enable Brotli for text-based responses
9. THE Cloudflare configuration SHALL enable HTTP/2 and HTTP/3 protocols
10. WHERE a .glb file is requested, THE CDN SHALL serve it with Content-Type: model/gltf-binary

### Requirement 3: 3D Asset Pipeline

**User Story:** As a content manager, I want automated deployment of 3D assets to CDN, so that character models load quickly for travelers worldwide.

#### Acceptance Criteria

1. THE 3D asset source files SHALL be stored in packages/assets-3d directory
2. WHEN a .glb file is added to packages/assets-3d, THEN THE build pipeline SHALL compress it using gltf-transform with draco and meshopt
3. THE compressed .glb file SHALL be at least 40% smaller than the original file
4. WHEN a 3D asset changes in packages/assets-3d, THEN THE GitHub Actions workflow SHALL upload the compressed version to Cloudflare CDN
5. THE uploaded asset URL SHALL include a hash-based version identifier for cache busting
6. THE 3D asset SHALL be accessible at https://app.vitale.vn/assets/characters/{character-id}-{hash}.glb
7. WHEN a 4G mobile client requests a character GLB file, THEN THE file SHALL load within 3 seconds
8. THE Character GLB file SHALL be smaller than 500KB after compression
9. THE texture files referenced by GLB SHALL use JPEG format with quality 85% or WebP format
10. WHERE a character has multiple animation clips, THE GLB SHALL include all clips in a single file

### Requirement 4: Backend Clean Architecture Structure

**User Story:** As a backend developer, I want a Clean Architecture project structure, so that domain logic remains independent of infrastructure concerns and the codebase is maintainable.

#### Acceptance Criteria

1. THE backend SHALL have four separate .NET projects: ViTale.Domain, ViTale.Application, ViTale.Infrastructure, ViTale.WebApi
2. THE ViTale.Domain project SHALL have zero external dependencies except .NET standard libraries
3. THE ViTale.Application project SHALL reference only ViTale.Domain
4. THE ViTale.Infrastructure project SHALL reference ViTale.Domain and ViTale.Application
5. THE ViTale.WebApi project SHALL reference ViTale.Application and ViTale.Infrastructure
6. THE ViTale.Domain project SHALL contain entities: Traveler, PassportAccount, Product, Character, StoryChapter, Checkpoint, CheckinRecord, Stamp, Badge, Voucher, Partner, ChatSession, ChatMessage
7. THE ViTale.Domain project SHALL contain value objects: QrCode, GeoCoordinate
8. THE ViTale.Domain project SHALL contain domain events: CheckpointUnlockedEvent, ProductActivatedEvent, AccountLinkedEvent
9. THE ViTale.Application project SHALL contain use case interfaces: IProductRepository, IAiChatService, IAuthenticationService, ICheckpointService
10. THE ViTale.Infrastructure project SHALL contain concrete implementations of all Application interfaces
11. THE ViTale.WebApi Program.cs SHALL configure dependency injection for all services
12. WHEN ViTale.WebApi starts, THEN THE DI container SHALL read configuration from environment variables

### Requirement 5: Domain Model - Traveler and Authentication

**User Story:** As a platform architect, I want a traveler entity supporting both anonymous and registered users, so that we can implement lazy registration without forcing account creation.

#### Acceptance Criteria

1. THE Traveler entity SHALL have properties: Id (Guid), AnonymousId (string), LinkedAccountId (Guid?), Preferences (JSONB), CreatedAt (DateTime)
2. THE PassportAccount entity SHALL have properties: Id (Guid), OAuthProvider (enum: Apple, Google), OAuthUserId (string), Email (string), CreatedAt (DateTime)
3. WHERE LinkedAccountId is null, THE Traveler SHALL be considered anonymous
4. WHERE LinkedAccountId is not null, THE Traveler SHALL be considered registered
5. THE Preferences column SHALL store JSON data with keys: preferredLanguage, notificationsEnabled
6. WHEN a new Traveler is created without OAuth, THEN THE AnonymousId SHALL be a unique 12-character alphanumeric string
7. THE Traveler.Id SHALL use sequential GUID generation for database performance
8. FOR ALL Travelers, THE CreatedAt timestamp SHALL be stored in UTC
9. THE database SHALL enforce unique constraint on PassportAccount.OAuthUserId per OAuthProvider
10. THE database SHALL enforce foreign key constraint between Traveler.LinkedAccountId and PassportAccount.Id

### Requirement 6: Domain Model - Products and Characters

**User Story:** As a product manager, I want to track physical QR products and their activation status, so that each wool doll or passport cover can be uniquely identified and linked to a regional character.

#### Acceptance Criteria

1. THE Product entity SHALL have properties: Id (Guid), QRCode (unique string), ProductType (enum: Doll, PassportCover), Region (string), ActivatedAt (DateTime?), ActivatedByTravelerId (Guid?)
2. THE Character entity SHALL have properties: Id (Guid), Name (string), Region (string), ModelUrl (string), AnimationClips (JSONB), Description (string)
3. THE QRCode value object SHALL validate format: alphanumeric, 16 characters minimum, 32 characters maximum
4. THE database SHALL enforce unique constraint on Product.QRCode
5. WHERE Product.ActivatedAt is null, THE Product SHALL be considered unactivated
6. WHERE Product.ActivatedAt is not null, THE Product SHALL be considered activated
7. THE Region property SHALL use ISO 3166-2 subdivision codes (e.g., VN-HN for Hanoi)
8. THE Character.ModelUrl SHALL reference a CDN path to the GLB file
9. THE Character.AnimationClips JSONB SHALL map Action_Tags to animation clip names (e.g., {"WAVE": "Wave_Anim", "SMILE": "Smile_Anim"})
10. THE database SHALL enforce foreign key constraint between Product.ActivatedByTravelerId and Traveler.Id

### Requirement 7: Domain Model - Story and Checkpoints

**User Story:** As a content designer, I want to define story chapters unlocked by visiting checkpoints, so that travelers experience location-based cultural narratives.

#### Acceptance Criteria

1. THE StoryChapter entity SHALL have properties: Id (Guid), Title (string), ContentKey (string), Region (string), UnlockCondition (JSONB), SortOrder (int)
2. THE Checkpoint entity SHALL have properties: Id (Guid), Name (string), Latitude (decimal), Longitude (decimal), Radius (int), StoryChapterId (Guid?), Region (string), IsActive (bool)
3. THE CheckinRecord entity SHALL have properties: Id (Guid), TravelerId (Guid), CheckpointId (Guid), CheckinAt (DateTime), ClientGeneratedId (Guid), SyncStatus (enum: Pending, Synced)
4. THE GeoCoordinate value object SHALL validate: Latitude between -90 and 90, Longitude between -180 and 180
5. THE Checkpoint.Radius SHALL be specified in meters, with minimum value 10 and maximum value 1000
6. THE UnlockCondition JSONB SHALL specify required checkpoint IDs as array: {"requiredCheckpointIds": ["guid1", "guid2"]}
7. THE StoryChapter.ContentKey SHALL reference a key in the Translations table for i18n support
8. WHERE CheckinRecord.SyncStatus is Pending, THE record SHALL be created offline and awaiting upload
9. WHERE CheckinRecord.SyncStatus is Synced, THE record SHALL be confirmed by the server
10. THE database SHALL enforce unique constraint on (TravelerId, CheckpointId, ClientGeneratedId) to prevent duplicate checkins
11. THE database SHALL enforce foreign key constraints: CheckinRecord.TravelerId → Traveler.Id, CheckinRecord.CheckpointId → Checkpoint.Id, Checkpoint.StoryChapterId → StoryChapter.Id

### Requirement 8: Domain Model - Gamification

**User Story:** As a game designer, I want to define stamps and badges with achievement conditions, so that travelers are motivated to explore more locations and complete specific patterns.

#### Acceptance Criteria

1. THE Stamp entity SHALL have properties: Id (Guid), TravelerId (Guid), CheckpointId (Guid), ImageUrl (string), EarnedAt (DateTime)
2. THE Badge entity SHALL have properties: Id (Guid), Name (string), Description (string), ImageUrl (string), ConditionType (enum), ConditionValue (JSONB)
3. THE TravelerBadge entity SHALL have properties: TravelerId (Guid), BadgeId (Guid), EarnedAt (DateTime), composite primary key (TravelerId, BadgeId)
4. THE ConditionType enum SHALL include values: CheckpointCount, RegionComplete, VoucherCount, ConsecutiveDays, PartnerVisit
5. WHERE ConditionType is CheckpointCount, THE ConditionValue SHALL specify: {"count": 5}
6. WHERE ConditionType is RegionComplete, THE ConditionValue SHALL specify: {"region": "VN-HN"}
7. WHERE ConditionType is VoucherCount, THE ConditionValue SHALL specify: {"count": 3, "category": "food"}
8. WHERE ConditionType is ConsecutiveDays, THE ConditionValue SHALL specify: {"days": 7}
9. WHERE ConditionType is PartnerVisit, THE ConditionValue SHALL specify: {"partnerIds": ["guid1", "guid2"]}
10. THE database SHALL enforce foreign key constraints: Stamp.TravelerId → Traveler.Id, Stamp.CheckpointId → Checkpoint.Id, TravelerBadge.TravelerId → Traveler.Id, TravelerBadge.BadgeId → Badge.Id
11. THE database SHALL enforce unique constraint on (TravelerId, CheckpointId) for Stamp to prevent duplicate stamps

### Requirement 9: Domain Model - Vouchers and Partners

**User Story:** As a business development manager, I want to manage partner businesses and their vouchers, so that travelers receive discounts and partners gain customer traffic.

#### Acceptance Criteria

1. THE Partner entity SHALL have properties: Id (Guid), Name (string), Type (enum: Restaurant, Hotel, TourOperator), ContactEmail (string), PhoneNumber (string), Address (string), Latitude (decimal), Longitude (decimal), PriorityScore (int), IsActive (bool)
2. THE Voucher entity SHALL have properties: Id (Guid), PartnerId (Guid), Title (string), Description (string), DiscountType (enum: Percentage, FixedAmount), DiscountValue (decimal), MinimumSpend (decimal?), MaxRedemptions (int?), ValidFrom (DateTime), ValidUntil (DateTime), IsActive (bool)
3. THE TravelerVoucher entity SHALL have properties: Id (Guid), TravelerId (Guid), VoucherId (Guid), ClaimedAt (DateTime), RedeemedAt (DateTime?), RedemptionCode (string)
4. THE Partner.Type enum SHALL include values: Restaurant, Hotel, TourOperator, Museum, Shop
5. THE Voucher.DiscountType enum SHALL include values: Percentage, FixedAmount, FreeItem
6. THE Partner.PriorityScore SHALL range from 0 to 100, with higher values appearing first in recommendations
7. WHERE Voucher.ValidUntil < current_timestamp, THE Voucher SHALL be considered expired
8. WHERE TravelerVoucher.RedeemedAt is null, THE voucher SHALL be considered claimed but not used
9. WHERE TravelerVoucher.RedeemedAt is not null, THE voucher SHALL be considered fully redeemed
10. THE RedemptionCode SHALL be a unique 8-character alphanumeric code generated when voucher is claimed
11. THE database SHALL enforce foreign key constraints: Voucher.PartnerId → Partner.Id, TravelerVoucher.TravelerId → Traveler.Id, TravelerVoucher.VoucherId → Voucher.Id
12. THE database SHALL enforce unique constraint on TravelerVoucher.RedemptionCode

### Requirement 10: Domain Model - Chat System

**User Story:** As an AI integration specialist, I want to track chat sessions and messages with metadata, so that we can provide contextual AI responses and implement conversation summarization.

#### Acceptance Criteria

1. THE ChatSession entity SHALL have properties: Id (Guid), TravelerId (Guid), StartedAt (DateTime), LastMessageAt (DateTime), TurnCount (int), CondensedContext (string?), CurrentCheckpointId (Guid?)
2. THE ChatMessage entity SHALL have properties: Id (Guid), SessionId (Guid), Role (enum: User, Assistant), Content (string), AudioUrl (string?), ActionTags (string array), CreatedAt (DateTime)
3. THE ChatMessage.Role enum SHALL include values: User, Assistant, System
4. THE ActionTags array SHALL contain strings matching Character animation clip keys (e.g., ["WAVE", "SMILE"])
5. WHEN a new ChatMessage is added, THE ChatSession.TurnCount SHALL increment by 1
6. WHEN a new ChatMessage is added, THE ChatSession.LastMessageAt SHALL update to current timestamp
7. THE ChatSession.CondensedContext SHALL store summarized conversation history when TurnCount exceeds threshold
8. THE database SHALL enforce foreign key constraints: ChatSession.TravelerId → Traveler.Id, ChatSession.CurrentCheckpointId → Checkpoint.Id, ChatMessage.SessionId → ChatSession.Id
9. THE database SHALL index ChatSession.TravelerId and ChatMessage.SessionId for query performance
10. FOR ALL ChatMessages, THE CreatedAt timestamp SHALL be stored in UTC

### Requirement 11: Database Schema and Migrations

**User Story:** As a database administrator, I want a versioned schema with migrations, so that database changes are tracked and can be applied consistently across environments.

#### Acceptance Criteria

1. THE ViTale_Platform SHALL use Entity Framework Core for database migrations
2. THE initial migration SHALL create all tables: Travelers, PassportAccounts, Products, Characters, StoryChapters, Checkpoints, CheckinRecords, Stamps, Badges, TravelerBadges, Partners, Vouchers, TravelerVouchers, ChatSessions, ChatMessages, Translations
3. THE Translations table SHALL have columns: Id (Guid), LanguageCode (string), ContentKey (string), ContentValue (text), composite unique constraint on (LanguageCode, ContentKey)
4. THE Translations table SHALL use JSONB column type for PostgreSQL
5. WHEN the api container starts, THE migration command "dotnet ef database update" SHALL run automatically
6. THE migration SHALL complete within 30 seconds for an empty database
7. WHERE a migration fails, THE api container SHALL log the error and exit with non-zero status
8. THE seed data SHALL insert 15 Checkpoint records for Hanoi region (VN-HN)
9. THE seed data SHALL insert 1 Character record for Hanoi region with sample animation clips
10. THE seed data SHALL insert sample StoryChapters linked to the Hanoi checkpoints
11. THE seed data SHALL insert sample Badge definitions for common achievements
12. THE database SHALL use btree indexes on: Traveler.AnonymousId, Product.QRCode, Checkpoint(Latitude, Longitude), CheckinRecord(TravelerId, CheckinAt)

### Requirement 12: Authentication Middleware - Anonymous Identity

**User Story:** As a user, I want to use the app immediately without creating an account, so that I can start exploring before deciding to register.

#### Acceptance Criteria

1. WHEN a client makes the first request without a vitale_session cookie, THEN THE authentication middleware SHALL generate a new Anonymous_ID
2. THE Anonymous_ID SHALL be a cryptographically secure random string of 12 alphanumeric characters
3. WHEN the Anonymous_ID is generated, THE middleware SHALL create a new Traveler record with AnonymousId populated
4. THE middleware SHALL set an HTTP-Only cookie named vitale_session with value containing the Traveler.Id
5. THE cookie SHALL have attributes: Domain=.vitale.vn, SameSite=Lax, Secure=true, Max-Age=31536000
6. WHEN a subsequent request includes the vitale_session cookie, THE middleware SHALL validate and extract the Traveler.Id
7. WHERE the Traveler.Id in the cookie is invalid, THE middleware SHALL generate a new Anonymous_ID and Traveler
8. THE middleware SHALL attach the authenticated Traveler object to the HTTP context for use by controllers
9. FOR ALL requests to /api/auth/link-account, THE middleware SHALL allow anonymous travelers
10. FOR ALL requests to /api/chat/* endpoints, THE middleware SHALL require a valid vitale_session cookie
11. THE middleware SHALL execute within 5 milliseconds per request

### Requirement 13: Authentication - Account Linking

**User Story:** As a traveler who has been using the app anonymously, I want to link my Apple or Google account, so that my progress is permanently saved and accessible across devices.

#### Acceptance Criteria

1. THE API SHALL expose endpoint POST /api/v1/auth/link-account accepting JSON body: {oAuthProvider: string, oAuthToken: string}
2. WHEN /auth/link-account is called with valid Apple oAuthToken, THEN THE API SHALL verify the token with Apple's identity service
3. WHEN /auth/link-account is called with valid Google oAuthToken, THEN THE API SHALL verify the token with Google's identity service
4. WHERE the oAuthToken is invalid, THE API SHALL return HTTP 401 with error message "Invalid OAuth token"
5. WHERE the oAuthToken is valid, THE API SHALL extract the user's email and OAuth user ID
6. WHERE a PassportAccount already exists with the same OAuthProvider and OAuthUserId, THE API SHALL return HTTP 409 with error message "Account already linked to another user"
7. WHERE no PassportAccount exists, THE API SHALL create a new PassportAccount record
8. THE API SHALL update the current anonymous Traveler's LinkedAccountId to reference the new PassportAccount
9. THE API SHALL preserve all existing CheckinRecords, Stamps, TravelerBadges, and TravelerVouchers associated with the anonymous Traveler
10. WHEN account linking succeeds, THE API SHALL generate a JWT containing claims: TravelerId, Email, IsRegistered=true
11. THE JWT SHALL be signed with the JWT_SECRET from environment configuration
12. THE JWT SHALL have expiration time of 7 days from issuance
13. THE API SHALL return HTTP 200 with JSON body: {success: true, jwt: string, traveler: object}
14. THE account linking operation SHALL complete within 2 seconds under normal network conditions

### Requirement 14: JWT Refresh Flow

**User Story:** As a registered user, I want my session to remain valid without frequent re-authentication, so that I have a seamless experience across multiple days.

#### Acceptance Criteria

1. THE API SHALL expose endpoint POST /api/v1/auth/refresh accepting no body parameters
2. WHEN /auth/refresh is called with a valid JWT in the vitale_session cookie, THE API SHALL verify the JWT signature
3. WHERE the JWT is expired but was issued within the last 30 days, THE API SHALL issue a new JWT with fresh 7-day expiration
4. WHERE the JWT is expired and was issued more than 30 days ago, THE API SHALL return HTTP 401 with error message "Session expired, please log in again"
5. WHERE the JWT signature is invalid, THE API SHALL return HTTP 401 with error message "Invalid token"
6. WHEN a new JWT is issued, THE API SHALL update the vitale_session cookie with the new JWT value
7. THE refresh operation SHALL complete within 100 milliseconds
8. THE JWT SHALL include claim: IssuedAt (timestamp) for calculating token age

### Requirement 15: Product Activation API

**User Story:** As a traveler who scanned a QR code on a wool doll, I want to activate the product and see my character, so that I can start my cultural journey.

#### Acceptance Criteria

1. THE API SHALL expose endpoint POST /api/v1/products/activate accepting JSON body: {qrCode: string}
2. WHEN /products/activate is called with a valid QRCode, THE API SHALL look up the Product record
3. WHERE no Product exists with the provided QRCode, THE API SHALL return HTTP 404 with error message "Product not found"
4. WHERE the Product.ActivatedAt is not null, THE API SHALL return HTTP 409 with error message "Product already activated"
5. WHERE the Product is unactivated, THE API SHALL update Product.ActivatedAt to current timestamp
6. THE API SHALL update Product.ActivatedByTravelerId to the current Traveler.Id from authentication context
7. THE API SHALL look up the Character associated with the Product.Region
8. THE API SHALL fire domain event ProductActivatedEvent containing ProductId and TravelerId
9. WHEN activation succeeds, THE API SHALL return HTTP 200 with JSON body: {travelerId: guid, character: {id, name, region, modelUrl, animationClips}}
10. THE activation operation SHALL be atomic (all database updates succeed or all fail)
11. THE activation endpoint SHALL complete within 500 milliseconds
12. FOR ALL valid activations, THE operation SHALL be idempotent if called multiple times with the same QRCode by the same Traveler

### Requirement 16: Passport Status API

**User Story:** As a traveler, I want to view my digital passport showing stamps, badges, and story progress, so that I can track my journey achievements.

#### Acceptance Criteria

1. THE API SHALL expose endpoint GET /api/v1/passport/me requiring authentication
2. WHEN /passport/me is called, THE API SHALL query all Stamps for the current Traveler
3. THE API SHALL query all Badges earned by the current Traveler via TravelerBadge join
4. THE API SHALL query all StoryChapters unlocked for the current Traveler's visited checkpoints
5. THE API SHALL calculate completion percentage per region: (visited checkpoints / total checkpoints) * 100
6. THE API SHALL return HTTP 200 with JSON body containing: {stamps: array, badges: array, unlockedStories: array, regionProgress: object, totalCheckpoints: int, visitedCheckpoints: int}
7. THE stamps array SHALL include checkpoint information: {stampId, checkpointName, imageUrl, earnedAt}
8. THE badges array SHALL include: {badgeId, name, description, imageUrl, earnedAt}
9. THE unlockedStories array SHALL include: {chapterId, title, contentKey, region}
10. THE regionProgress object SHALL map region codes to percentages: {"VN-HN": 60, "VN-SG": 20}
11. THE endpoint SHALL complete within 200 milliseconds for travelers with up to 100 stamps
12. WHERE the Traveler has no stamps or badges, THE API SHALL return empty arrays with HTTP 200

### Requirement 17: Checkin Submission API (Batch Support)

**User Story:** As a traveler who visited multiple checkpoints while offline, I want to upload all my checkins when I regain connectivity, so that my passport is updated with all my visits.

#### Acceptance Criteria

1. THE API SHALL expose endpoint POST /api/v1/checkins accepting JSON body: {checkins: array}
2. THE checkins array SHALL contain objects with: {checkpointId: guid, checkinAt: ISO8601 timestamp, clientGeneratedId: guid, latitude: number, longitude: number}
3. WHEN /checkins is called with an empty array, THE API SHALL return HTTP 400 with error message "Checkins array cannot be empty"
4. FOR ALL checkins in the array, THE API SHALL validate that the CheckpointId exists in the database
5. WHERE a CheckpointId is invalid, THE API SHALL skip that checkin and log a warning but continue processing others
6. FOR ALL valid checkins, THE API SHALL verify that (latitude, longitude) is within the Checkpoint.Radius
7. WHERE coordinates are outside the radius, THE API SHALL skip that checkin and include in response: {invalidCheckins: array}
8. FOR ALL valid checkins, THE API SHALL check for existing CheckinRecord with same (TravelerId, CheckpointId, ClientGeneratedId)
9. WHERE a matching record exists, THE API SHALL skip duplicate insertion (idempotent behavior)
10. WHERE no matching record exists, THE API SHALL insert new CheckinRecord with SyncStatus=Synced
11. FOR ALL new checkins at a checkpoint, THE API SHALL create a corresponding Stamp record
12. WHEN all checkins are processed, THE API SHALL evaluate badge unlock conditions for the Traveler
13. WHERE a badge condition is newly satisfied, THE API SHALL insert TravelerBadge record and include in response
14. THE API SHALL evaluate story chapter unlock conditions based on the Traveler's complete checkin history
15. WHERE a story chapter condition is newly satisfied, THE API SHALL fire CheckpointUnlockedEvent
16. WHEN processing completes, THE API SHALL return HTTP 200 with JSON body: {processedCount: int, skippedCount: int, newBadges: array, unlockedStories: array, invalidCheckins: array}
17. THE batch processing SHALL handle up to 50 checkins in a single request
18. THE endpoint SHALL complete within 2 seconds for a batch of 50 checkins
19. THE checkin processing SHALL be transactional: WHERE any database error occurs, ALL checkins in the batch SHALL roll back

### Requirement 18: Nearby Checkpoints API

**User Story:** As a traveler exploring a city, I want to see nearby checkpoints on a map, so that I know where to go next for story content.

#### Acceptance Criteria

1. THE API SHALL expose endpoint GET /api/v1/checkpoints/nearby accepting query parameters: lat (decimal), lng (decimal), radius (optional int, default 5000)
2. WHERE lat or lng parameters are missing, THE API SHALL return HTTP 400 with error message "Latitude and longitude are required"
3. WHERE lat is outside range -90 to 90, THE API SHALL return HTTP 400 with error message "Invalid latitude"
4. WHERE lng is outside range -180 to 180, THE API SHALL return HTTP 400 with error message "Invalid longitude"
5. THE API SHALL use PostGIS or equivalent spatial query to find Checkpoints within the specified radius in meters
6. THE API SHALL calculate distance from (lat, lng) to each Checkpoint using haversine formula
7. THE API SHALL return only Checkpoints where IsActive=true
8. THE API SHALL order results by distance ascending (nearest first)
9. THE API SHALL limit results to maximum 20 checkpoints
10. FOR ALL returned checkpoints, THE API SHALL include whether the current Traveler has already checked in
11. WHEN query succeeds, THE API SHALL return HTTP 200 with JSON body: {checkpoints: array}
12. THE checkpoints array SHALL contain: {checkpointId, name, latitude, longitude, distance, isVisited, storyChapterTitle, region}
13. THE endpoint SHALL complete within 300 milliseconds
14. WHERE no checkpoints exist within radius, THE API SHALL return empty array with HTTP 200

### Requirement 19: Partner Recommendations API

**User Story:** As a traveler looking for a place to eat, I want recommendations for nearby restaurants, so that I can find good food and earn vouchers.

#### Acceptance Criteria

1. THE API SHALL expose endpoint GET /api/v1/partners/recommendations accepting query parameters: type (optional enum), lat (optional decimal), lng (optional decimal)
2. WHERE type parameter is provided, THE API SHALL filter Partners by Partner.Type matching the value
3. WHERE type parameter is invalid, THE API SHALL return HTTP 400 with error message "Invalid partner type"
4. WHERE lat and lng parameters are provided, THE API SHALL calculate distance to each Partner
5. WHERE lat and lng are provided, THE API SHALL order results by distance ascending
6. WHERE lat and lng are NOT provided, THE API SHALL order results by PriorityScore descending
7. THE API SHALL return only Partners where IsActive=true
8. THE API SHALL limit results to maximum 30 partners
9. FOR ALL returned partners, THE API SHALL include count of available vouchers
10. WHEN query succeeds, THE API SHALL return HTTP 200 with JSON body: {partners: array}
11. THE partners array SHALL contain: {partnerId, name, type, address, latitude, longitude, distance?, availableVouchers: int, priorityScore}
12. THE endpoint SHALL complete within 400 milliseconds

### Requirement 20: Chat Message API

**User Story:** As a traveler, I want to ask questions to the AI assistant and receive text, audio, and character animations, so that I get an immersive cultural storytelling experience.

#### Acceptance Criteria

1. THE API SHALL expose endpoint POST /api/v1/chat/message accepting JSON body: {sessionId: guid?, message: string, currentCheckpointId: guid?} or multipart form with audio file
2. WHERE sessionId is null or invalid, THE API SHALL create a new ChatSession for the current Traveler
3. WHERE sessionId is valid, THE API SHALL load the existing ChatSession
4. WHEN a new ChatSession is created, THE API SHALL set StartedAt to current timestamp and TurnCount to 0

## Notes on Property-Based Testing Recommendations

### Requirements Using Round-Trip Properties (Always Test These!)

- **Requirement 58**: Parsers and serializers for JSON, dates, QR codes, coordinates
  - Critical pattern: `parse(format(x)) == x` and `deserialize(serialize(obj)) == obj`
  - Parsers are tricky and round-trip testing catches edge cases

### Requirements Using Idempotency Properties

- **Requirement 59**: Checkin submission and product activation
  - Pattern: `f(x) = f(f(x))` - doing operation twice equals doing it once
  - Essential for offline-first sync to prevent duplicates

### Requirements Using Invariants

- **Requirement 60**: Gamification mechanics (stamps, badges, story unlocks)
  - Properties that must always hold: stamp count = distinct checkins, badge timestamps logical, etc.
  - Monotonicity: progress never decreases

### Requirements Using Model-Based Testing

- **Requirement 61**: Checkpoint radius validation
  - Compare optimized implementation against simple reference model
  - Verify complex distance calculations match specification

### Requirements NOT Suitable for Property-Based Testing

The following should use integration tests with 1-3 examples instead:

- Infrastructure configuration (Docker, DNS, CDN) - binary success/failure
- External service integration (OAuth, cloud APIs) - testing provider behavior, not our code
- Deployment and CI/CD - deterministic workflows
- Health checks and monitoring - simple alive/dead checks
- Database migrations - run once, verify schema
